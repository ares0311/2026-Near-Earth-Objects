# Alert Pathway Guide

Technical reference for the alert-pathway decision tree implemented in `alert.py`.

---

## Overview

The alert pathway governs how a scored NEO candidate is reported to the wider
astronomical community.  It is a **mandatory, ordered gate sequence** — no step
may be skipped.  The pipeline never autonomously asserts an impact probability;
all public hazard communication is deferred to MPC and CNEOS.

---

## Alert Pathways

| Pathway | Meaning |
|---|---|
| `internal_candidate` | Below all reporting thresholds; kept in internal log only |
| `known_object` | Matched to MPC catalog; no new submission needed |
| `mpc_submission` | Meets detection thresholds; submit to MPC for confirmation |
| `neocp_followup` | Object placed on NEOCP; request independent observations |
| `nasa_pdco_notify` | High-confidence PHA; follow NASA PDCO notification protocol |

---

## Gate Conditions

A candidate must satisfy **all** of the following to advance beyond
`internal_candidate`:

1. `real_bogus_score ≥ 0.65` (configurable; default in `detect.py`)
2. Not matched to MPC known-object catalog within 5 arcsec
3. Tracklet spans ≥ 2 observations on ≥ 2 distinct nights

To advance to `mpc_submission`:

1. `real_bogus_score ≥ 0.65`
2. Arc length ≥ 1 day (orbit quality code ≥ 2)
3. Motion rate consistent with solar system object (0.05–60 arcsec/hr)

To advance to `nasa_pdco_notify` (the highest gate):

1. Computed MOID ≤ 0.05 AU
2. Orbit quality code ≥ 2 (multi-night arc)
3. Tier 1 `real_bogus_score ≥ 0.90`
4. NOT matched to MPC known object
5. Independent confirmation received (≥ 2 observatories or ≥ 24 h on NEOCP)
6. CNEOS Scout/Sentry assigns impact probability ≥ 0.01%

## Step-by-Step Protocol

### Step 1 — MPC Submission

Submit a standard MPC 80-column observation report or MPC JSON format to:

```
https://minorplanetcenter.net/iau/info/ObsFormat.html
```

Use `format_mpc_report(neo)` or `draft_mpc_submission(neo)` from `alert.py`.
The cover letter generated by `generate_mpc_cover_letter(neo)` includes the
mandatory guardrail statement.

**Guardrail**: Every MPC submission must include the statement:
> "Do NOT publicly announce any impact probability."

### Step 2 — NEOCP Monitoring

After MPC submission, monitor the NEO Confirmation Page for independent
confirmations:

- Wait at least **24 hours** or until **≥ 2 independent observatory
  confirmations** are received.
- Use `monitor_neocp(neo, timeout_hours=24)` in `alert.py` for automated
  polling.
- If the object is not confirmed, demote to `internal_candidate` and log.

### Step 3 — NASA PDCO Notification

Triggered only when CNEOS Scout or Sentry assigns an impact probability ≥ 0.01%.

1. Open a GitHub Issue tagged `[HAZARD-ALERT]` with full provenance.
2. Generate a structured alert package via `generate_alert_package(neo)`.
3. Send report to:
   - **NASA PDCO**: https://www.nasa.gov/planetarydefense/contact
   - **IAU CBAT**: https://www.cbat.eps.harvard.edu/
4. **Do NOT** quote or publish any impact probability in any public channel.
   Defer all public communication to NASA/CNEOS.

## MPC 80-Column Format

Each observation line is exactly 80 characters, following the MPC standard:

```
Columns  Content
1–5      Packed MPC designation or blank
6        Discovery asterisk (*)
7        Note 1 (observation type)
8        Note 2 (observer + telescope code)
9–17     Date (YYYY MM DD.ddddd)
18–31    RA  (HH MM SS.ddd)
32–44    Dec (±DD MM SS.dd)
45–56    Blank (for standard astrometry)
57–65    Magnitude
66–67    Filter band
68–70    Blank
71–77    MPC observatory code (left-justified, right-padded)
```

Use `format_mpc_observation(obs, designation, obs_code)` to generate
individual lines, and `format_mpc_report(neo)` to generate a complete report.

For the newer MPC JSON format use `format_mpc_json(neo)`.

---

## ADES PSV Format

The ADES (Astrometry Data Exchange Standard) format is MPC's modern replacement
for the 80-column format.  Use `format_mpc_ades_psv(neo)` to generate a
pipe-separated-value ADES submission.

---

## Validation

Before submitting, call `ready_for_submission(neo)` to validate all gate
conditions.  It returns `(bool, list[str])` — the boolean indicates whether
submission is safe; the list contains any unmet conditions.

Use `validate_alert_package(package)` to confirm a generated alert package
contains all required fields and a guardrail statement containing "NOT".

---

## Alert Package Structure

A complete alert package (from `generate_alert_package(neo)`) contains:

| Key | Content |
|---|---|
| `neo_id` | Pipeline object identifier |
| `alert_pathway` | Selected pathway string |
| `moid_au` | Computed MOID in AU (None if unknown) |
| `mpc_report` | Full MPC 80-column report text |
| `mpc_json` | MPC JSON format dict |
| `submission_checklist` | Gate-condition checklist text |
| `cover_letter` | Formal MPC cover letter |
| `guardrail_statement` | Mandatory "NOT" statement |
| `impact_notification` | PDCO notification dict (nasa_pdco_notify only) |

---

## Guardrails (Non-Negotiable)

- **Never skip** MPC submission and independent confirmation before Step 3.
- **Never quote** an impact probability in any public output.
- **Never suppress** a genuine alert out of uncertainty — report to MPC and
  let authorities assess.
- Store full provenance (observations, orbit fit, MOID computation) with every
  alert action.
- Every guardrail statement must contain the word **"NOT"** (validated by
  `validate_alert_package`).
